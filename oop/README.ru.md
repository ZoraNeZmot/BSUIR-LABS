# ООТПиСП — лабораторные работы (порядковый номер в группе 12)

Каждая лабораторная работа представляет собой самостоятельный
Go-модуль с собственным `go.mod` и графическим интерфейсом на Fyne.
Лабораторные строятся одна на основе другой, но исходный код
полностью продублирован, чтобы каждую можно было открыть, проверить
и сдать независимо.

| Папка | Тема                                  | Вариант |
|-------|---------------------------------------|---------|
| `lab3` | Сериализация объектов                 | 3 — Текст |
| `lab4` | Плагины — иерархия                    | —       |
| `lab5` | Плагины — функциональность            | 3 — Шифрование |
| `lab6` | Паттерны                              | —       |

Вариант определяется по формуле
`(порядковый_номер mod количество_вариантов) + 1`. Для номера
**12** и пяти вариантов получается **3** в обеих работах, где это
требуется.

## Готовые сборки

В папке `bin/` лежат уже собранные `*.exe` для каждой работы
вместе со всеми необходимыми ресурсами (папки `plugins/` и
`funcplugins/`). Подробнее — `bin/README.ru.md`.

## Сборка из исходников

Для каждой лабораторной выполните одинаковую последовательность:

```
cd labN
go mod tidy
go run ./cmd/app
```

Первый `go mod tidy` скачает Fyne (~150 МБ один раз); следующие
лабораторные используют общий кеш Go-модулей.

## Краткое содержание лабораторных

* **lab3** — иерархия из 9 классов транспорта (Car / Motorcycle /
  Truck / Boat / Ship / Airplane / Helicopter, плюс общие базовые
  классы Land / Water / Air). Текстовый формат сериализации; редактор
  и сериализатор работают через `[]FieldDescriptor`, поэтому
  добавление нового класса **не требует** `if` / `switch` / рефлексии
  по конкретному типу.
* **lab4** — динамическая загрузка новых классов из JSON-дескрипторов
  в папке `plugins/`. Положили файл — нажали «Reload plugins» — новый
  тип сразу появился в GUI. В комплекте идут демо-плагины
  `bicycle.json` и `submarine.json`.
* **lab5** — вариант 3, шифрование. Три обратимых алгоритма
  (`xor`, `caesar`, `aes-cfb`) регистрируются через `init()`; их
  экземпляры с конкретными параметрами описываются JSON-плагинами в
  папке `funcplugins/`. Меню «Plugin settings…» включает/выключает
  плагины и редактирует параметры (на 10 баллов).
* **lab6** — три явные демонстрации паттернов:
  * **Адаптер** — `internal/funcplugins/algos/checksum_adapter.go`
    оборачивает «плагин товарища» (`internal/friendlib`,
    вариант 5 — *сохранение контрольной суммы*), приводя его API с
    sidecar-строкой к интерфейсу `Algorithm`, который ожидает один
    `[]byte`.
  * **Одиночка (Singleton)** — `internal/appsettings.Instance()`
    возвращает единственный для процесса набор настроек.
  * **Наблюдатель (Observer)** — `internal/events.Bus` позволяет
    списку, индикатору изменений и журналу аудита независимо
    подписываться на события.

## Соответствие требованиям

* Комментарии в коде написаны по-английски (требование правил).
* Каждая работа лежит в своей папке.
* Нет ни одного `if` / `switch` / рефлексии, ветвящегося по
  конкретному классу транспорта — и сериализатор, и редактор полностью
  обобщённые.
* Тесты:
  * `lab3`: `go test ./internal/storage/...` — round-trip всех
    встроенных классов через текстовый формат.
  * `lab6`: `go test ./internal/funcplugins/...` — round-trip всей
    цепочки `caesar -> xor -> aes -> friend-checksum` и проверка
    того, что адаптер контрольной суммы обнаруживает повреждение
    данных.
