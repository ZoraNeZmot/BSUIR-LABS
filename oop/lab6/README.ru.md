# Лабораторная 6 — Паттерны

Развивает лабораторную 5. Иерархия классов транспорта, JSON-плагины
и конвейер шифрования полностью унаследованы. Лабораторная 6
добавляет три явные демонстрации шаблонов проектирования.

## 1. Адаптер — `internal/friendlib` + `internal/funcplugins/algos/checksum_adapter.go`

«Плагин товарища» представлен независимым пакетом
`internal/friendlib`, который реализует **вариант 5** из лабораторной 5
— *сохранение контрольной суммы*. Его API **намеренно несовместим** с
тем, что ожидает хост:

```go
// Чего хочет хост (один []byte на вход, один на выход).
type Algorithm interface {
    Encode(data []byte, params map[string]string) ([]byte, error)
    Decode(data []byte, params map[string]string) ([]byte, error)
}

// Что предлагает товарищ (sidecar-метаданные, нет params).
type ChecksumKeeper struct{ Algorithm string }
func (k *ChecksumKeeper) Wrap(payload []byte) (out []byte, sidecar string, err error)
func (k *ChecksumKeeper) Verify(payload []byte, sidecar string) error
```

`checksumAdapter` оборачивает `*friendlib.ChecksumKeeper` и переводит
все детали:

* инлайнит sidecar-строку в полезную нагрузку как заголовок
  `FRIENDCHK:<hex>:`, чтобы один срез байт нёс обе части;
* достаёт нужный параметр из `map[string]string` хоста и кладёт его в
  поле структуры товарища;
* реализует четыре метода `Algorithm` и регистрируется в том же
  глобальном реестре, что и встроенные алгоритмы — поэтому в GUI он
  выглядит как любой другой плагин.

Почему адаптер уместен: ни код хоста (конвейер уже зафиксирован), ни
код товарища (это «сторонняя» библиотека) менять нельзя. Адаптер —
канонический способ соединить два готовых контракта.

## 2. Одиночка (Singleton) — `internal/appsettings`

`appsettings.Instance()` возвращает единственный для процесса
`*Settings`, в котором хранятся имя пользователя (показывается в
заголовке окна), путь до последнего открытого файла (используется в
«Load default») и две папки плагинов.

Почему уместен Singleton: у этих настроек нет естественного
владельца, читают их сразу несколько слоёв (GUI, загрузчик плагинов,
сохранение/загрузка), а тащить один и тот же `*Settings` через все
конструкторы — это лишний шум. `sync.Once` гарантирует ленивую
инициализацию ровно один раз, а внутренний `sync.RWMutex` делает
параллельное чтение безопасным.

## 3. Наблюдатель (Observer) — `internal/events`

`events.Bus` — это маленький синхронный pub/sub. UI подписывает на
него **трёх независимых наблюдателей**:

1. главный список и подписи в шапке перерисовываются при любом
   событии;
2. флажок «есть несохранённые изменения» поднимается/сбрасывается и
   отображается в заголовке окна;
3. журнал аудита внизу окна append-only фиксирует каждое изменение.

Почему уместен Observer: у каждого подписчика своя задача и свой
жизненный цикл. При прямых вызовах места изменения должны были бы
знать про каждого из них; через шину новые подписчики добавляются и
отключаются без правок издателей.

## Запуск

```
cd lab6
go mod tidy
go run ./cmd/app
```

Готовая сборка: `bin/lab6/lab6.exe` (рядом лежат папки `plugins/` и
`funcplugins/`, включая `friend-checksum.json`).

Включите в «Plugin settings…» одновременно
`Friend's checksum (adapted)` и `AES-256 (strong)`, чтобы увидеть
полный конвейер (шифрование → контрольная сумма) при сохранении;
загрузка прогонит файл через обратную цепочку (проверка хеша →
расшифровка). Если файл повреждён, адаптер контрольной суммы
сообщит об этом и не позволит загрузить мусор.
